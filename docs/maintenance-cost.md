# ZiWei 可维护性与运行成本评估

## 1. 当前结论

这个仓库当前的**运行成本很低**，因为：

- CLI/skill 本地运行，不依赖数据库服务
- 地理解析走本地 JSON，不需要在线 API 请求
- 测试耗时短，当前 `npm test` 约 **3.8s**，`npm run test:cli` 约 **4.6s**
- 地理主数据 `cn-district-full.json` 当前约 **535 KB**，规模仍处于“直接内存加载完全可接受”的区间

但它的**维护风险不在运行时，而在数据构建阶段**：

- 上游阿里云 DataV 公共源有 `403/498` 风控
- 当前全量库仅 **1159** 条记录，明显不是全国完整区县覆盖
- `cn-district-full.meta.json` 已记录大量跳过项，说明“构建成功”不等于“覆盖完整”

一句话判断：

> 现在适合把它当作**可离线运行的原型库**，不适合当作“自动稳定全量更新”的生产级全国行政区划底座。

---

## 2. 数据体积与仓库成本

### 2.1 当前观测

- 仓库目录总大小：约 **9.8 MB**
- `node_modules/`：约 **8.3 MB**
- `data/`：约 **548 KB**
- `data/geo/cn-district-full.json`：约 **535 KB**
- `data/geo/cn-district-min.json`：约 **6.9 KB**

### 2.2 解释

当前仓库真正需要长期关注的，不是数据文件本身，而是：

1. **不要把 `node_modules/` 误提交到仓库**
   - 它远大于业务数据
   - 好在 `.gitignore` 已忽略

2. **JSON 主数据仍足够小**
   - 500KB 级别即使每次进程启动同步读入，成本也很低
   - 对当前 CLI / skill 场景，没有必要为了这点数据引入 Redis、SQLite 服务或远程缓存

3. **后续增长空间仍充足**
   - 即便补齐到数千区县记录，JSON 预计也只是数 MB 级
   - 真正触发架构升级的信号，不会是“文件略大”，而会是“别名爆炸 / 历史区划映射 / 多来源合并 / 高频 API 服务化”

---

## 3. 构建成本评估

## 3.1 当前构建路径

当前通过 `scripts/build-cn-admin-db.js` 从公共 GeoJSON 源批量抓取，生成：

- `data/geo/cn-district-full.json`
- `data/geo/cn-district-full.meta.json`

## 3.2 成本与风险

### 低成本部分

- 脚本逻辑简单，维护门槛低
- 构建产物是纯静态 JSON，方便审查与版本管理
- 不依赖数据库迁移或额外部署系统

### 高风险部分

- 上游接口存在限流/风控，当前 meta 中已出现大量 `403/498`
- 并发默认值 `12` 对公共源略激进，容易导致构建不完整
- 当前流程更像“尽量抓全”，还不是“可验证完整性”的稳定流水线

## 3.3 判断

当前脚本适合：

- 初次拉全
- 人工触发式更新
- 做数据调研与补丁构建

不适合：

- CI 每次自动全量跑
- 用户请求时在线即时重建
- 把“脚本成功退出”当成“数据已完整”

---

## 4. 运行时是否需要缓存

## 4.1 当前结论：**不需要额外缓存层**

原因：

- `resolver.js` 在模块加载时一次性读入 JSON
- 数据规模仅几十万字节级
- 当前使用方式是单次 CLI / skill 调用，不是高 QPS 服务

这意味着当前已经天然具备“进程内缓存”特征：

- 单个进程内只读一次
- 后续解析直接走内存数组

## 4.2 什么时候再考虑更强缓存

只有在以下场景出现时才值得升级：

- 同一进程里会有大量连续查询
- 别名规模明显扩大，扫描数组开始变慢
- 需要多版本数据并存对比
- 需要跨进程共享缓存或服务化部署

在那之前，**新增 Redis / 持久缓存层只会增加维护成本，不会带来成比例收益**。

## 5. 版本标识与生成元数据建议

这是当前最值得补强的部分。

## 5.1 为什么要加强 meta

目前 `cn-district-full.meta.json` 已有：

- `generatedAt`
- `source`
- `stats`
- `skipped`

但还不够回答下面这些维护问题：

- 这份数据到底属于哪个版本？
- 文件内容是否变了？
- 构建时用了什么并发参数？
- 这次构建体积多大，是否异常膨胀？
- 失败项主要是哪些 HTTP 状态？

## 5.2 已补的小改动

已对 `scripts/build-cn-admin-db.js` 补充元数据生成能力，后续重建时会额外写入：

- `datasetVersion`
- `build.durationMs`
- `build.concurrency`
- `artifacts.json.bytes`
- `artifacts.json.gzipBytes`
- `artifacts.json.sha256`
- `stats.skippedByStatus`
- `maintenance.recommendedRefresh`
- `maintenance.cachePolicy`

这能显著改善：

- 构建审计
- 版本对比
- 数据异常排查
- 后续自动化验收

## 5.3 下一步建议

建议把下面几条作为验收口径：

1. **无版本号的构建产物，不进入主分支**
2. **跳过项数量异常升高时，人工复核后再提交**
3. **sha256 变化但记录数几乎不变时，也要看 diff，防止静默污染**
4. **至少记录一次“推荐刷新周期”与“上游已知缺陷”**

## 6. 后续更新维护策略

推荐采用“三层维护”模式。

## 6.1 第 1 层：稳定运行层

默认运行时只使用仓库内静态数据：

- `cn-district-full.json`
- `cn-district-min.json`
- `cn-district-full.meta.json`

原则：

- 用户请求时不碰公网源
- 线上只读，不在线重建
- 让排盘能力稳定优先于数据“理论最新”

## 6.2 第 2 层：人工更新层

按以下节奏触发：

- 年度一次正式校对
- 重大区划调整后人工补丁
- 高频用户地名命中失败时补 alias / min 数据

建议更新步骤：

1. 降低并发重跑构建，如 `GEO_CONCURRENCY=4`
2. 对比新旧 meta：记录数、跳过项、sha256、体积
3. 观察跳过项是否集中在某些省份
4. 用回归测试 + CLI 验收测试确认没有打坏现有行为
5. 再决定是否合入主分支

## 6.3 第 3 层：未来升级层

只有当以下需求出现时，再考虑改造：

- 引入 SQLite
- 增加历史区划映射
- 做拼音/简称/旧称索引
- 多来源合并校验（统计局 / 民政部 / 社区库）

也就是说：

> 先把“离线 JSON + 明确 meta + 人工补丁”做稳，再谈数据库化。

---

## 7. 是否需要进一步代码改造

### 现在建议做

- 保持 `resolver.js` 现有进程内加载方式
- 保持 JSON 为主存储
- 继续强化构建元数据
- 在 README 或 docs 中明确“这是一份离线缓存产物，不是实时全量官方库”

### 现在不建议做

- 不建议加 Redis
- 不建议加在线回源兜底
- 不建议把构建放到高频 CI
- 不建议为了 500KB 数据过早切 SQLite

---

## 8. 建议优先级

### P1（建议立刻执行）

- 使用增强后的 meta 机制重新生成一次数据文件
- 以后提交数据时同时提交对应 meta
- README 中补充维护说明与更新策略

### P2（近期可做）

- 为 `meta` 增加“最低记录数阈值”验收规则
- 为若干高频省市补充人工兜底数据
- 把 `403/498` 的高发地区整理成补丁清单

### P3（后续再看）

- 引入 SQLite 导出脚本
- 接入国家统计局年度代码表做校验
- 维护历史区划/旧称映射

---

## 9. 一句话建议

**这个项目当前最该做的不是“加缓存”，而是“把离线数据的版本、构建元数据和人工更新策略做扎实”。**

这样运行成本仍然低，但维护可见性会高很多。
