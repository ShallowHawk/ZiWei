# ziwei-doushu

先完成稳定排盘，再决定是否扩写解读。

如需面向用户输出解读文案，先遵守 `../../docs/limitations.md` 的风险限制：不要把盘面摘要写成宿命化、恐吓式或重大事项的确定结论。

## 1. 输入要求

优先收集以下信息：

1. 公历日期：`YYYY-MM-DD`
2. 时间：`HH:mm`
3. 性别：`male` / `female`（也兼容 `男` / `女`）
4. 出生地：尽量精确到区县级

若用户只提供城市、省份或模糊地名，也可以先尝试解析；一旦脚本返回歧义候选，就继续追问，不要自行猜测。

如果用户能直接提供经纬度，可跳过地点解析，直接使用 `--longitude` 与 `--latitude`。

## 2. 工作流

1. 收集日期、时间、性别、出生地。
2. 对中国大陆常见地点，尽量追问到区县级；对模糊地点，先运行脚本看是否能唯一解析。
3. 调用脚本获取结构化结果。
4. 先阅读 `trueSolar`、`chart`、`palaces`、`highlights` 等核心字段。
5. 若用户只要排盘结果，直接返回文本或 JSON。
6. 若用户要更长的分析，先阅读 `../../docs/limitations.md` 确认风险边界，再结合 `references/ziwei-knowledge.md` 做二次解读。

## 3. 调用方式

### 按地点排盘

```bash
node skills/ziwei-doushu/scripts/ziwei.js --date 1990-05-15 --time 14:30 --gender male --place 北京市东城区
```

### 按经纬度排盘

```bash
node skills/ziwei-doushu/scripts/ziwei.js --date 1990-05-15 --time 14:30 --gender female --longitude 121.4737 --latitude 31.2304 --place 上海
```

### 输出 JSON

```bash
node skills/ziwei-doushu/scripts/ziwei.js --date 1988-02-09 --time 23:40 --gender female --place 杭州市余杭区 --json
```

若地点存在歧义，脚本会报错并给出候选；补全到更精确地点后再重试。

## 4. 如何解读输出

默认文本输出适合直接回复用户，JSON 输出适合继续交给上层模型或工作流处理。

重点关注：

- `location`：地点解析结果与坐标来源
- `trueSolar`：真太阳时、时辰、`timeIndex`
- `chart`：命宫、身宫、农历、干支等核心盘面信息
- `palaces`：十二宫结构化数据
- `highlights`：命宫、财帛、官禄、夫妻、迁移、福德等重点摘要
- `extensions`：扩展结构；已含本命四化、各宫大限，传入 `--targetDate` 后还可得到大限/流年最小结构

如果要写自然语言解读，优先基于结构化字段展开，不要脱离盘面自由发挥；尤其不要把健康、寿命、疾病、婚姻、投资收益等内容写成确定性判断。

## 5. 资源说明

- 理论参考：`references/ziwei-knowledge.md`
- 地理最小数据集：`../../data/geo/cn-district-min.json`
- 入口脚本：`scripts/ziwei.js`

只有在需要补充命理解释口径时，再读取理论参考；排盘本身以脚本输出为准。

## 6. 注意事项

- 真太阳时已同时纳入经度修正与时间方程。
- 子时必须区分：
  - 早子时 `timeIndex=0`
  - 晚子时 `timeIndex=12`
- 当前地理数据是“全国可运行最小版”，不是全国全量区县库。
- 遇到地点歧义时，要继续追问用户，不要替用户拍板。
- 不要重写 `iztro` 的排盘逻辑；工程重点应放在输入清洗、时间修正、地点解析、结果组织和上层解读衔接。

## 7. 能力边界

这个 skill 当前擅长：

- 稳定排盘
- 输出结构化结果
- 为上层解读提供可靠输入

这个 skill 当前不保证：

- 全国任意地点都能直接解析
- 自动生成完整专业咨询级长报告
- 覆盖所有命理流派的深度解释差异

在对外描述时，把它当作“排盘与结构化结果 skill”，而不是“全自动命理大师系统”。
